Data Flow Control with Passant

Data Flow Control (DFC) lets applications express policies over how database records may be derived, transformed, aggregated, and written. Passant enforces those policies by rewriting SQL before it runs, so agents can keep using useful data while the database layer rejects unsafe flows.

Install See the API

Why DFC

Access control can decide whether an agent may read a table, but many safety requirements depend on what the agent does after reading it. A tax agent may need receipt data to prepare deductions, but every inserted expense should be grounded in an actual receipt and comply with deduction rules. A reporting agent may read sensitive rows, but public outputs should aggregate across enough users before release.

DFC policies describe these derivation rules in PGN. Passant registers the policies on a wrapped database connection and transforms SQL so policy checks run with the query itself. Violating rows can be removed with ON FAIL REMOVE, while severe violations can abort with ON FAIL KILL.

  • Policy language: PGN source, sink, dimension, constraint, and resolution clauses.
  • Engine: Rust-backed SQL rewriting exposed through a Python API.
  • Backends: DuckDB first, with adapters for SQLite, PostgreSQL, ClickHouse, DataFusion, and Umbra.

Install

Install the published Python package, then import passant in your code.

Examples

Law-abiding

Remove deductions that violate tax rules

This policy applies when a query derives Expenses from Receipts. Meal expenses with business use above 50 percent fail the constraint and are removed from the insert.

Python 
import duckdb
from passant import Policy, dfc

raw = duckdb.connect()
conn = dfc(raw)

conn.execute("CREATE TABLE Receipts (id INTEGER, biz_use INTEGER, cat VARCHAR)")
conn.execute("CREATE TABLE Expenses (id INTEGER, biz_use INTEGER, cat VARCHAR)")
conn.execute("""
    INSERT INTO Receipts VALUES
    (1, 80, 'Meal'),
    (2, 30, 'Meal'),
    (3, 80, 'Travel')
""")

conn.register_policy(
    Policy.from_pgn("""
        SOURCE Receipts SINK Expenses
        CONSTRAINT Expenses.biz_use <= 50 OR Receipts.cat != 'Meal'
        ON FAIL REMOVE
    """)
)

conn.execute("""
    INSERT INTO Expenses (id, biz_use, cat)
    SELECT Receipts.id, Receipts.biz_use, Receipts.cat
    FROM Receipts
""")

rows = conn.fetchall("SELECT id, biz_use, cat FROM Expenses ORDER BY id")
assert rows == [(2, 30, "Meal"), (3, 80, "Travel")]
Private

Only release aggregates over multiple users

Public reports should not expose a statistic derived from one user. This policy keeps output groups only when more than one distinct receipt owner contributed, or when a one-row current-user dimension says the session is admin.

Python 
import duckdb
from passant import Policy, dfc

raw = duckdb.connect()
conn = dfc(raw)

conn.execute("CREATE TABLE Receipts (uid INTEGER, cat VARCHAR, amount INTEGER)")
conn.execute("CREATE TABLE CurrentUser (is_admin BOOLEAN)")
conn.execute("""
    INSERT INTO Receipts VALUES
    (1, 'Meal', 25),
    (2, 'Meal', 40),
    (3, 'Travel', 180)
""")
conn.execute("INSERT INTO CurrentUser VALUES (FALSE)")

conn.register_policy(
    Policy.from_pgn("""
        SOURCE Receipts
        DIMENSION CurrentUser
        CONSTRAINT count(distinct Receipts.uid) > 1
            OR max(CurrentUser.is_admin)
        ON FAIL REMOVE
    """)
)

rows = conn.fetchall("""
    SELECT cat, SUM(amount) AS total
    FROM Receipts
    GROUP BY cat
    ORDER BY cat
""")

assert rows == [("Meal", 65)]
Grounded

Abort fabricated writes

SOURCE REQUIRED means writes into the sink must actually query the source. Here, every inserted expense must be derived from a receipt with the same id. A fabricated expense aborts the query.

Python 
import duckdb
from passant import Policy, dfc

raw = duckdb.connect()
conn = dfc(raw)

conn.execute("CREATE TABLE Receipts (id INTEGER, item VARCHAR)")
conn.execute("CREATE TABLE Expenses (id INTEGER, item VARCHAR)")
conn.execute("INSERT INTO Receipts VALUES (1, 'coffee')")

conn.register_policy(
    Policy.from_pgn("""
        SOURCE REQUIRED Receipts SINK Expenses
        CONSTRAINT Receipts.id = Expenses.id
        ON FAIL KILL
    """)
)

conn.execute("""
    INSERT INTO Expenses (id, item)
    SELECT Receipts.id, Receipts.item
    FROM Receipts
""")

try:
    conn.execute("""
        INSERT INTO Expenses (id, item)
        SELECT 99, 'phantom'
        FROM Receipts
    """)
except Exception as exc:
    print(f"Blocked fabricated expense: {exc}")
Flock extension

Use models only for targeted classification

DFC does not ask a model to decide whether the whole data flow is safe. Instead, the policy can call a narrowly scoped SQL function in CONSTRAINT, such as Flock llm_filter, and keep the rest of the policy deterministic.

Note: Passant does not ship a built-in LLM(...) function. Install the Flock DuckDB extension and use its scalar functions in policy constraints. Optional execution tests require OPENAI_API_KEY or FLOCK_OPENAI_API_KEY.

Python 
import duckdb
from passant import Policy, Resolution, dfc

raw = duckdb.connect()
raw.execute("INSTALL flock FROM community")
raw.execute("LOAD flock")

conn = dfc(raw)
conn.execute("CREATE TABLE Receipts (id INTEGER, biz_use INTEGER, cat VARCHAR)")
conn.execute("CREATE TABLE Expenses (id INTEGER, biz_use INTEGER, cat VARCHAR)")
conn.execute("""
    INSERT INTO Receipts VALUES
    (1, 80, 'client dinner'),
    (2, 30, 'client dinner'),
    (3, 80, 'taxi')
""")

conn.register_policy(
    Policy.from_pgn("""
        SOURCE Receipts SINK Expenses
        CONSTRAINT llm_filter(
            {'model_name': 'default'},
            {'prompt': 'Is this a meal?', 'context_columns': [{'data': Receipts.cat}]}
        ) OR Expenses.biz_use <= 50
        ON FAIL REMOVE
    """)
)

conn.execute("""
    INSERT INTO Expenses (id, biz_use, cat)
    SELECT Receipts.id, Receipts.biz_use, Receipts.cat
    FROM Receipts
""")

rows = conn.fetchall("SELECT id, biz_use, cat FROM Expenses ORDER BY id")
assert rows == [(2, 30, "client dinner"), (3, 80, "taxi")]
